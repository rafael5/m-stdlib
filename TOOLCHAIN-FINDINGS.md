m-stdlib development surfaces is logged here, against m-cli,
tree-sitter-m, or m-standard. The post-Phase-1 publication gate for
m-cli and tree-sitter-m is "no open P0/P1 entries in this file."

| 2026-05-05 | **P1** | m-stdlib | `STDASSERT.raises` `$ETRAP` arg-less `quit` | When an XECUTE'd error fires inside an extrinsic-function chain (e.g. `$$f^STDFMT → $$render → $$expand → $$apply → $$convert  set $ECODE=...`), the trap handler `set captured=$ecode set $ecode="" quit` runs in the routine where the error fired. Its arg-less `QUIT` is illegal in extrinsic context and triggers `%YDB-E-NOTEXTRINSIC` (M17), which cascades into `Z150374554` and clobbers `captured` away from the original user code. STDFMT's six error-path tests (UNCLOSED-BRACE, UNESCAPED-RBRACE, UNKNOWN-TYPE, MISSING-ARG ×3) all hit this, as do STDDATETST and STDCSVTST in-progress (independently observed: `actual: ,M17,Z150374554,`). STDB64 / STDUUID don't surface this because they have no `raises` calls; STDASSERT's own raises-tests trigger errors in **procedure** context (`set x=1/0`). | **resolved 2026-05-06**: `raises` now captures `raisesLvl=$zlevel` at $ETRAP-set time and the trap unwinds via `zgoto raisesLvl:raisesUnwound^STDASSERT` instead of arg-less `quit`. ZGOTO unwinds the M-stack to the captured level cleanly regardless of intervening extrinsic frames, so the trap no longer needs to satisfy procedure-vs-extrinsic context constraints at the firing site. Verified by `tFormatInvalidRaises` in STDLOGTST (raises-driven test that previously crashed the harness; now passes 48/48 in the suite). The deferred STDFMT / STDDATE / STDCSV error-path tests should now be re-enableable on their next sweep. |
| 2026-05-05 | P2 | m-cli / YottaDB | `$GET($ECODE)` silently crashes the YDB process | Calling `$GET($ECODE)` (or any `$GET` of an intrinsic special variable) raises a fatal YDB error that bypasses any user `$ETRAP`, exits the YDB process with code 253, and emits **nothing** to stderr or stdout. From `m test`'s perspective the suite simply "fails" with 0/0 assertions and no diagnostic — the runner reports `FAIL <suite> (0/0 passed)` with no FAIL line, no expected/actual pair, no traceback. Burned ~30 minutes during L5 STDDATE bisecting which `if $get($ecode)'="" quit ""` line was the culprit; the routine compiles and lints clean, the smoke-test version of each function works, but the moment the offending line is reached the process dies. Workaround: use bare `$ECODE'=""` — `$ECODE` is always defined (defaults to `""`) so `$GET` is never needed. Easy to write by reflex from `$GET(local)` habit. | open — two improvements possible: (1) **m-cli lint**: add a rule that flags `$GET($X)` for any intrinsic special variable `$X` (always wrong, since intrinsics are always defined). (2) **m-cli runner**: when the YDB child exits non-zero with empty stdout/stderr, surface the exit code and a "process crashed before producing output" diagnostic instead of reporting `0/0 passed`. (3) **YDB**: `$GET($ECODE)` should be a clean compile-time or runtime error with a recognizable code, not a silent crash. |
| 2026-05-05 | P2 | YottaDB | `$ETRAP` that `TROLLBACK`s its own scope then re-raises `$ECODE` does not propagate the error to the next outer `$ETRAP` | Surfaced building L8 STDFIX. Pattern: routine `R` does `tstart` to open a scope, then runs user code that sets `$ECODE`. `R`'s own `$ETRAP` fires, saves `$ecode`, clears it, runs `trollback target` to undo the local scope, sets `$ecode=saved` to re-raise, and `quit`s the trap. By documented YDB semantics, when `$ETRAP` exits with `$ECODE` non-empty, control should unwind frame-by-frame until another trap is found or the process exits. **Observed**: the rollback DOES happen (verified by global-write rollback assertions) and `R`'s frame unwinds, but the outer caller's `$ETRAP` never fires — `$ECODE` arrives at the caller as empty. Reproduced in pure procedure-form (no extrinsic chain), so this is distinct from the M17 NOTEXTRINSIC P1 above. Happens for STDFIX `with`/`invoke`, and the same shape causes the deferred error-path tests in STDFMT, STDDATE, STDCSV. Hypothesis: YDB may treat `$ETRAP`-driven `TROLLBACK` of a frame-local transaction as an implicit "the trap consumed the error" signal, suppressing further unwind regardless of `$ECODE` state at trap-quit. Either documented-but-non-obvious or a defect; minimal repro should be filed upstream. | open — workaround: don't rely on re-raise from a trap that rolls back its own transaction. STDFIX ships the rollback + scope-cleanup observables verified, with the explicit re-raise contract documented but unverified by automated test. Same deferred footprint in STDFMT (6 tests), STDDATE, STDCSV. Worth a minimal-repro upstream report to YottaDB to confirm intentional vs defect. |

Reproduced on `STDASSERTTST::tEqPassesEqual`, `STDJSONTST::tParseNull`, `STDJSONTST::tTypeOnUndefinedReturnsEmpty`, `STDMOCKTST::tResolveReturnsTargetWhenNoMock`. Output shape — exit 253, nothing to stderr — is identical to the documented `$GET($ECODE)` silent-crash signature (P2 row above), so the regression is most likely a single line in the new single-test wrapper that touches an intrinsic via `$GET`, or that re-uses a path the whole-suite codepath bypasses. Reported as `m test: 1 suite(s), 0 passed, 1 failed, 0/0 assertions passed` with no diagnostic, exactly the runner-blind-spot pattern flagged in the P2 above. | open — needs in-process stderr capture in m-cli (the runner is currently swallowing the YDB process's last gasp). | 2026-05-05 | P3 | m-cli | `m test` default-isolation throughput on high-test-count suites | Default `m test` wraps each test in `tstart` / user code / `trollback $tlevel-1` via STDFIX. Per-test overhead is small individually but compounds: STDJSONTST (65 tests) under default isolation produced 53 sequential `ok` lines within a 300 s window before the operator's external timeout fired; the same 65 tests with `--no-isolation` complete much faster. Operationally fine, but two related observations worth filing: (a) running the harness under `timeout N m test ... 2>&1 \| tee file` interacts badly with timeout's SIGTERM — the `tee` target ends up empty even when the suite was actively producing TAP, suggesting m-cli's stdio is block-buffered when stdout is a pipe rather than a TTY (line-buffered would let `tee` survive a kill); (b) there is no per-suite timeout option in the `m test` CLI, so users must rely on external `timeout` and live with the buffering issue. | open — two small improvements: (1) line-buffer m-cli's TAP/text formatter output (or pass `-u` / call `sys.stdout.flush()` after each suite event) so external timeout / tee / head pipelines preserve partial output. (2) document the `--no-isolation` perf characteristic in `m test --help` and recommend it for >50-test suites until per-test STDFIX overhead is amortised. |
| 2026-05-06 | **P1** | m-stdlib | `$$encode^STDJSON` recursive descent — distinct from the (resolved) STDASSERT.raises P1 above | Surfaced re-enabling the deferred STDLOG-JSON-emission tests after the STDASSERT.raises ZGOTO fix. | 2026-05-05 | docs | YottaDB / m-stdlib | `tstart`/`trollback` must balance per routine frame (TPQUIT) | **Not a YDB defect — documented design.** YDB enforces that a routine cannot `quit` with an unbalanced `tstart` (raises `%YDB-E-TPQUIT, Cannot QUIT out of a routine with an active transaction`). This means the typical Python-style fixture API (`setup() opens scope; later teardown() closes it`) is structurally impossible in YDB — the matching `trollback`/`tcommit` must live in the same routine call as the `tstart`. Hit this writing L8 STDFIX: the orchestration-plan §6.4 sketch described `SETUP^STDFIX(tag)` / `TEARDOWN^STDFIX(tag)` as separate procedures, which compiles and parses fine but blows up at runtime. Logged here for posterity so future contributors don't try the same. Standard MUMPS (ANSI X11.1) permits cross-frame transactions; YDB's stricter rule is a deliberate guardrail against transaction leaks. | resolved (design-time) — STDFIX now exposes only one-shot wrappers (`with(tag,code)` / `invoke(tag,code)`), which open AND close the scope in the same frame. The runner protocol consumes `with`/`invoke` instead of `setup`/`teardown`. Documented in `docs/modules/stdfix.md` and the orchestration-plan §6.4 should be updated to match. |
