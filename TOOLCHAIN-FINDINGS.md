# Toolchain findings — m-stdlib regression log

Per the implementation plan §14: every toolchain weakness that
m-stdlib development surfaces is logged here, against m-cli,
tree-sitter-m, or m-standard. The post-Phase-1 publication gate for
m-cli and tree-sitter-m is "no open P0/P1 entries in this file."

| Date | Severity | Project | Module | Finding | Status |
|---|---|---|---|---|---|
| 2026-04-30 | P1 | m-cli | `src/m_cli/test/runner.py` | `run_case` (single-test selection via `m test FILE.m::tLabel`) hard-codes `do start^TESTRUN(.pass,.fail)` and `do report^TESTRUN(pass,fail)` (lines 168–173). Whole-suite execution works against STDASSERT-driven suites because the output-protocol parser is shape-based, but per-case selection breaks. | open — companion PR scheduled when m-cli adapts to STDASSERT (plan §13.5). m-stdlib v0.0.1 only uses whole-suite mode, so non-blocking. |
| 2026-04-30 | P3 | m-cli | `M-MOD-020` (by-reference args) | False positive when label name shadows formal parameter name (e.g. `pass(pass,desc)` inherited from `^TESTRUN` style). Lint-time analysis treats writes-to-formal as writes-to-local-of-same-name. Fires on the probe-stub STDASSERT. | **resolved (2026-04-30)** at v0.0.1 by renaming internal helpers to `recordPass` / `recordFail`; no m-cli change needed. |
| 2026-04-30 | P2 | m-cli | `M-MOD-020` (by-reference args) — test-suite idiom | False positive: test labels with `(pass,fail)` formals call `do eq^STDASSERT(.pass,.fail,...)` — m-cli's by-ref analysis can't follow into the called label and reports the formals as unwritten. Affects every `*TST.m` file. Workaround in v0.0.1: file-wide `; m-lint: disable-file=M-MOD-020` directive at the top of each TST. | open — m-cli could special-case test labels (name matches `t<UpperCase>(pass,fail)` and body delegates `.pass,.fail` to a known assertion library) to suppress automatically. |
| 2026-04-30 | P2 | m-cli | `M-MOD-024` (read-of-undefined-local) | Doesn't recognise `SET (a,b)=0` (M's parenthesised multi-assign) as defining `a` and `b`. Reports later reads of `a`/`b` as possibly-undefined. Workaround: write `SET a=0,b=0`. | open — small flow analysis fix; affects readability of test fixtures. |
| 2026-04-30 | P2 | YottaDB / m-cli | TRACE produces deep `FOR_LOOP` / `*CHILDREN` subscripts | `view "TRACE":1:"^ycov":""` records nested subscripts inside `FOR` loops (e.g. `^ycov("STDUUIDTST","tVerXYZ",N,"FOR_LOOP",1,"*CHILDREN",...)`). Tests with descriptive label names + FOR loops calling helpers blow the **default 64-byte** `KEY_SIZE` setting, raising `%YDB-E-GVSUBOFLOW` and corrupting captured data — making `m coverage` report 0% for the affected suite. | **resolved (2026-04-30)** by recreating the workspace DB with `KEY_SIZE=1019` + `BLOCK_SIZE=4096` in `tools/init-db.sh`. With the bump, full-corpus `make coverage` reports 22/22 labels (100%). Upstream improvement worth filing: m-cli could capture per-suite traces in separate processes to bound subscript depth, or pre-bump KEY_SIZE in its own setup helper. |
| 2026-05-05 | P2 | m-cli | `M-MOD-024` (read-of-undefined-local) — OPEN/CLOSE deviceparams | The linter parses YDB OPEN deviceparam lists like `open path:(readonly):5` or `open path:(newversion:stream:nowrap):5` (and CLOSE `close path:delete`) as expressions, treating tokens like `readonly`, `newversion`, `stream`, `nowrap`, and `delete` as local-variable reads. Fires M-MOD-024 on each, then cascades read-of-undefined complaints onto the formal parameters and other locals defined elsewhere in the routine (likely the misparse confuses control-flow analysis). Workaround in STDCSV v0.0.6: file-wide `; m-lint: disable-file=M-MOD-024` on the source and the test (the only files with file I/O so far). | open — m-cli should recognise YDB deviceparam keywords inside OPEN/USE/CLOSE expressions and skip the local-read analysis for them. |
| 2026-05-05 | **P1** | m-stdlib | `STDASSERT.raises` `$ETRAP` arg-less `quit` | When an XECUTE'd error fires inside an extrinsic-function chain (e.g. `$$f^STDFMT → $$render → $$expand → $$apply → $$convert  set $ECODE=...`), the trap handler `set captured=$ecode set $ecode="" quit` runs in the routine where the error fired. Its arg-less `QUIT` is illegal in extrinsic context and triggers `%YDB-E-NOTEXTRINSIC` (M17), which cascades into `Z150374554` and clobbers `captured` away from the original user code. STDFMT's six error-path tests (UNCLOSED-BRACE, UNESCAPED-RBRACE, UNKNOWN-TYPE, MISSING-ARG ×3) all hit this, as do STDDATETST and STDCSVTST in-progress (independently observed: `actual: ,M17,Z150374554,`). STDB64 / STDUUID don't surface this because they have no `raises` calls; STDASSERT's own raises-tests trigger errors in **procedure** context (`set x=1/0`). | open — fix: `$ETRAP` should use `ZGOTO N:label` or set a flag and `ZGOTO 0` to unwind cleanly out of arbitrary extrinsic depth. Alternative: ship a parallel `raisesx^STDASSERT` that is extrinsic-aware. STDFMT v0.0.3 documents the contract and ships error tests deferred. |
| 2026-05-05 | P2 | m-cli / YottaDB | `$GET($ECODE)` silently crashes the YDB process | Calling `$GET($ECODE)` (or any `$GET` of an intrinsic special variable) raises a fatal YDB error that bypasses any user `$ETRAP`, exits the YDB process with code 253, and emits **nothing** to stderr or stdout. From `m test`'s perspective the suite simply "fails" with 0/0 assertions and no diagnostic — the runner reports `FAIL <suite> (0/0 passed)` with no FAIL line, no expected/actual pair, no traceback. Burned ~30 minutes during L5 STDDATE bisecting which `if $get($ecode)'="" quit ""` line was the culprit; the routine compiles and lints clean, the smoke-test version of each function works, but the moment the offending line is reached the process dies. Workaround: use bare `$ECODE'=""` — `$ECODE` is always defined (defaults to `""`) so `$GET` is never needed. Easy to write by reflex from `$GET(local)` habit. | open — two improvements possible: (1) **m-cli lint**: add a rule that flags `$GET($X)` for any intrinsic special variable `$X` (always wrong, since intrinsics are always defined). (2) **m-cli runner**: when the YDB child exits non-zero with empty stdout/stderr, surface the exit code and a "process crashed before producing output" diagnostic instead of reporting `0/0 passed`. (3) **YDB**: `$GET($ECODE)` should be a clean compile-time or runtime error with a recognizable code, not a silent crash. |
| 2026-05-05 | P2 | YottaDB | `$ETRAP` that `TROLLBACK`s its own scope then re-raises `$ECODE` does not propagate the error to the next outer `$ETRAP` | Surfaced building L8 STDFIX. Pattern: routine `R` does `tstart` to open a scope, then runs user code that sets `$ECODE`. `R`'s own `$ETRAP` fires, saves `$ecode`, clears it, runs `trollback target` to undo the local scope, sets `$ecode=saved` to re-raise, and `quit`s the trap. By documented YDB semantics, when `$ETRAP` exits with `$ECODE` non-empty, control should unwind frame-by-frame until another trap is found or the process exits. **Observed**: the rollback DOES happen (verified by global-write rollback assertions) and `R`'s frame unwinds, but the outer caller's `$ETRAP` never fires — `$ECODE` arrives at the caller as empty. Reproduced in pure procedure-form (no extrinsic chain), so this is distinct from the M17 NOTEXTRINSIC P1 above. Happens for STDFIX `with`/`invoke`, and the same shape causes the deferred error-path tests in STDFMT, STDDATE, STDCSV. Hypothesis: YDB may treat `$ETRAP`-driven `TROLLBACK` of a frame-local transaction as an implicit "the trap consumed the error" signal, suppressing further unwind regardless of `$ECODE` state at trap-quit. Either documented-but-non-obvious or a defect; minimal repro should be filed upstream. | open — workaround: don't rely on re-raise from a trap that rolls back its own transaction. STDFIX ships the rollback + scope-cleanup observables verified, with the explicit re-raise contract documented but unverified by automated test. Same deferred footprint in STDFMT (6 tests), STDDATE, STDCSV. Worth a minimal-repro upstream report to YottaDB to confirm intentional vs defect. |
| 2026-05-05 | docs | YottaDB / m-stdlib | `tstart`/`trollback` must balance per routine frame (TPQUIT) | **Not a YDB defect — documented design.** YDB enforces that a routine cannot `quit` with an unbalanced `tstart` (raises `%YDB-E-TPQUIT, Cannot QUIT out of a routine with an active transaction`). This means the typical Python-style fixture API (`setup() opens scope; later teardown() closes it`) is structurally impossible in YDB — the matching `trollback`/`tcommit` must live in the same routine call as the `tstart`. Hit this writing L8 STDFIX: the orchestration-plan §6.4 sketch described `SETUP^STDFIX(tag)` / `TEARDOWN^STDFIX(tag)` as separate procedures, which compiles and parses fine but blows up at runtime. Logged here for posterity so future contributors don't try the same. Standard MUMPS (ANSI X11.1) permits cross-frame transactions; YDB's stricter rule is a deliberate guardrail against transaction leaks. | resolved (design-time) — STDFIX now exposes only one-shot wrappers (`with(tag,code)` / `invoke(tag,code)`), which open AND close the scope in the same frame. The runner protocol consumes `with`/`invoke` instead of `setup`/`teardown`. Documented in `docs/modules/stdfix.md` and the orchestration-plan §6.4 should be updated to match. |
