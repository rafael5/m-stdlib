# Toolchain findings — m-stdlib regression log

Per the implementation plan §14: every toolchain weakness that
m-stdlib development surfaces is logged here, against m-cli,
tree-sitter-m, or m-standard. The post-Phase-1 publication gate for
m-cli and tree-sitter-m is "no open P0/P1 entries in this file."

| Date | Severity | Project | Module | Finding | Status |
|---|---|---|---|---|---|
| 2026-04-30 | P1 | m-cli | `src/m_cli/test/runner.py` | `run_case` (single-test selection via `m test FILE.m::tLabel`) hard-codes `do start^TESTRUN(.pass,.fail)` and `do report^TESTRUN(pass,fail)` (lines 168–173). Whole-suite execution works against STDASSERT-driven suites because the output-protocol parser is shape-based, but per-case selection breaks. | open — companion PR scheduled when m-cli adapts to STDASSERT (plan §13.5). m-stdlib v0.0.1 only uses whole-suite mode, so non-blocking. |
| 2026-04-30 | P3 | m-cli | `M-MOD-020` (by-reference args) | False positive when label name shadows formal parameter name (e.g. `pass(pass,desc)` inherited from `^TESTRUN` style). Lint-time analysis treats writes-to-formal as writes-to-local-of-same-name. Fires on the probe-stub STDASSERT. | **resolved (2026-04-30)** at v0.0.1 by renaming internal helpers to `recordPass` / `recordFail`; no m-cli change needed. |
| 2026-04-30 | P2 | m-cli | `M-MOD-020` (by-reference args) — test-suite idiom | False positive: test labels with `(pass,fail)` formals call `do eq^STDASSERT(.pass,.fail,...)` — m-cli's by-ref analysis can't follow into the called label and reports the formals as unwritten. Affects every `*TST.m` file. Workaround in v0.0.1: file-wide `; m-lint: disable-file=M-MOD-020` directive at the top of each TST. | open — m-cli could special-case test labels (name matches `t<UpperCase>(pass,fail)` and body delegates `.pass,.fail` to a known assertion library) to suppress automatically. |
| 2026-04-30 | P2 | m-cli | `M-MOD-024` (read-of-undefined-local) | Doesn't recognise `SET (a,b)=0` (M's parenthesised multi-assign) as defining `a` and `b`. Reports later reads of `a`/`b` as possibly-undefined. Workaround: write `SET a=0,b=0`. | open — small flow analysis fix; affects readability of test fixtures. |
| 2026-04-30 | P2 | YottaDB / m-cli | TRACE produces deep `FOR_LOOP` / `*CHILDREN` subscripts | `view "TRACE":1:"^ycov":""` records nested subscripts inside `FOR` loops (e.g. `^ycov("STDUUIDTST","tVerXYZ",N,"FOR_LOOP",1,"*CHILDREN",...)`). Tests with descriptive label names + FOR loops calling helpers blow the **default 64-byte** `KEY_SIZE` setting, raising `%YDB-E-GVSUBOFLOW` and corrupting captured data — making `m coverage` report 0% for the affected suite. | **resolved (2026-04-30)** by recreating the workspace DB with `KEY_SIZE=1019` + `BLOCK_SIZE=4096` in `tools/init-db.sh`. With the bump, full-corpus `make coverage` reports 22/22 labels (100%). Upstream improvement worth filing: m-cli could capture per-suite traces in separate processes to bound subscript depth, or pre-bump KEY_SIZE in its own setup helper. |
| 2026-05-05 | **P1** | m-stdlib | `STDASSERT.raises` `$ETRAP` arg-less `quit` | When an XECUTE'd error fires inside an extrinsic-function chain (e.g. `$$f^STDFMT → $$render → $$expand → $$apply → $$convert  set $ECODE=...`), the trap handler `set captured=$ecode set $ecode="" quit` runs in the routine where the error fired. Its arg-less `QUIT` is illegal in extrinsic context and triggers `%YDB-E-NOTEXTRINSIC` (M17), which cascades into `Z150374554` and clobbers `captured` away from the original user code. STDFMT's six error-path tests (UNCLOSED-BRACE, UNESCAPED-RBRACE, UNKNOWN-TYPE, MISSING-ARG ×3) all hit this, as do STDDATETST and STDCSVTST in-progress (independently observed: `actual: ,M17,Z150374554,`). STDB64 / STDUUID don't surface this because they have no `raises` calls; STDASSERT's own raises-tests trigger errors in **procedure** context (`set x=1/0`). | open — fix: `$ETRAP` should use `ZGOTO N:label` or set a flag and `ZGOTO 0` to unwind cleanly out of arbitrary extrinsic depth. Alternative: ship a parallel `raisesx^STDASSERT` that is extrinsic-aware. STDFMT v0.0.3 documents the contract and ships error tests deferred. |
