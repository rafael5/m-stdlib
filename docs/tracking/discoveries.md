# Discoveries register

> **Note (2026-05-10).** This file was renamed from `TOOLCHAIN-FINDINGS.md`
> as part of the docs/tracking restructure. Scope expanded from
> "external toolchain bugs only" to "every discovery that wasn't
> anticipated in a locked plan but had to be addressed during
> implementation" — both internal m-stdlib design pivots and external
> findings against m-cli / tree-sitter-m / YottaDB / vista-meta.
> See [`README.md`](README.md) § Bucket 3 for the schema and the
> "what goes here" criteria.

This is the canonical register for issues that surfaced during
m-stdlib implementation but weren't anticipated in the locked plan.
Each row records the surprise, what the implementation did about it,
and the current status. Trackers in `docs/tracking/` and plans in
`docs/plans/` cite rows here when they record scope shifts.

**Severity legend.** `P0` blocks any release · `P1` blocks a phase
release · `P2` is a recurring nuisance / open workaround · `P3` is a
minor friction · `docs` is a documented design constraint, not a
defect.

**Subject legend.** `m-stdlib` (internal) · `m-cli` · `tree-sitter-m`
· `m-standard` · `YottaDB` · `vista-meta` · slash-pair if both.

**Plan impact.** What the implementation did (or will do) in response.
Required column. Empty `Plan impact` means the discovery doesn't
belong here — file it as an upstream issue instead.

**Publication gate** (carried over from the toolchain-findings
predecessor): m-cli and tree-sitter-m post-Phase-1 publication
requires "no open P0/P1 entries against those subjects."

| 2026-04-30 | P2 | YottaDB / m-cli | TRACE produces deep `FOR_LOOP` / `*CHILDREN` subscripts | `view "TRACE":1:"^ycov":""` records nested subscripts inside `FOR` loops (e.g. `^ycov("STDUUIDTST","tVerXYZ",N,"FOR_LOOP",1,"*CHILDREN",...)`). | 2026-05-05 | P2 | m-cli | `M-MOD-024` (read-of-undefined-local) — OPEN/CLOSE deviceparams | The linter parses YDB OPEN deviceparam lists like `open path:(readonly):5` or `open path:(newversion:stream:nowrap):5` (and CLOSE `close path:delete`) as expressions, treating tokens like `readonly`, `newversion`, `stream`, `nowrap`, and `delete` as local-variable reads. Cascades read-of-undefined complaints onto formal parameters and other locals defined elsewhere in the routine. | STDCSV v0.0.6 carries file-wide `; m-lint: disable-file=M-MOD-024` on the source and the test (the only files with file I/O at the time). Same workaround in any future m-stdlib module that opens a SEQ device until m-cli recognises YDB deviceparam keywords inside OPEN/USE/CLOSE expressions. | open |
| 2026-05-05 | **P1** | m-stdlib | `STDASSERT.raises` `$ETRAP` arg-less `quit` | When an XECUTE'd error fires inside an extrinsic-function chain (e.g. `$$f^STDFMT → $$render → $$expand → $$apply → $$convert  set $ECODE=...`), the trap handler `set captured=$ecode set $ecode="" quit` runs in the routine where the error fired. Its arg-less `QUIT` is illegal in extrinsic context and triggers `%YDB-E-NOTEXTRINSIC` (M17), which cascades into `Z150374554` and clobbers `captured` away from the original user code. STDFMT's six error-path tests (UNCLOSED-BRACE, UNESCAPED-RBRACE, UNKNOWN-TYPE, MISSING-ARG ×3) all hit this, as do STDDATETST and STDCSVTST in-progress. | STDASSERT.raises now captures `raisesLvl=$zlevel` at $ETRAP-set time and the trap unwinds via `zgoto raisesLvl:raisesUnwound^STDASSERT` instead of arg-less `quit`. ZGOTO unwinds the M-stack to the captured level cleanly regardless of intervening extrinsic frames. The deferred STDFMT / STDDATE / STDCSV error-path tests can now be re-enabled. | **resolved 2026-05-06** |
| 2026-05-05 | P2 | m-cli / YottaDB | `$GET($ECODE)` silently crashes the YDB process | Calling `$GET($ECODE)` (or any `$GET` of an intrinsic special variable) raises a fatal YDB error that bypasses any user `$ETRAP`, exits the YDB process with code 253, and emits **nothing** to stderr or stdout. From `m test`'s perspective the suite simply "fails" with 0/0 assertions and no diagnostic. Burned ~30 minutes during L5 STDDATE bisecting which `if $get($ecode)'="" quit ""` line was the culprit. | STDDATE rewrites to bare `$ECODE'=""` (no `$GET` — `$ECODE` is always defined). Convention propagated across m-stdlib. m-cli improvements open: (1) lint rule flagging `$GET($X)` for any intrinsic special variable; (2) runner surfaces non-zero exit + empty stdio as "process crashed before producing output" instead of `0/0 passed`; (3) YDB upstream — should be a clean error, not a silent crash. | open |
| 2026-05-05 | P2 | YottaDB | `$ETRAP` that `TROLLBACK`s its own scope then re-raises `$ECODE` does not propagate the error to the next outer `$ETRAP` | Surfaced building L8 STDFIX. Pattern: routine `R` does `tstart`, runs user code that sets `$ECODE`, `R`'s `$ETRAP` fires, saves `$ecode`, clears it, runs `trollback target`, sets `$ecode=saved` to re-raise, and `quit`s the trap. By documented YDB semantics, control should unwind frame-by-frame until another trap is found. **Observed**: rollback DOES happen but the outer caller's `$ETRAP` never fires — `$ECODE` arrives at the caller as empty. Hypothesis: YDB may treat `$ETRAP`-driven `TROLLBACK` of a frame-local transaction as an implicit "the trap consumed the error" signal. | STDFIX exposes only one-shot wrappers (`with(tag,code)` / `invoke(tag,code)`) that open AND close the scope in the same frame. The runner protocol consumes `with`/`invoke` instead of `setup`/`teardown`. STDFIX ships with rollback + scope-cleanup observables verified; the explicit re-raise contract is documented but unverified by automated test. Same deferred footprint in STDFMT (6 tests), STDDATE, STDCSV — re-enableable after the ZGOTO fix above. Worth a minimal-repro upstream report to YottaDB. | open |
| 2026-05-05 | **P1** | m-cli | `src/m_cli/test/runner.py` `run_case` (post-`23241a2`) | After the C1 fix that made `run_case` dispatch `do start^{protocol}` / `do report^{protocol}`, single-test mode (`m test FILE.m::tLabel`) now exits with **rc=253 and empty stdout/stderr** for every suite that loads STDASSERT. Reproduced on `STDASSERTTST::tEqPassesEqual`, `STDJSONTST::tParseNull`, `STDMOCKTST::tResolveReturnsTargetWhenNoMock`. Whole-suite mode of the same suites is healthy. Output shape — exit 253, nothing to stderr — matches the documented `$GET($ECODE)` silent-crash signature. | m-stdlib developers fall back to whole-suite reruns for iterative TDD on a single label. m-cli fix open: in-process stderr capture + investigation of regression introduced by the C1 protocol-routing change. | open |
| 2026-05-05 | P3 | m-cli | `m test` default-isolation throughput on high-test-count suites | Default `m test` wraps each test in `tstart` / user code / `trollback $tlevel-1` via STDFIX. Per-test overhead compounds: STDJSONTST (65 tests) under default isolation produced 53 sequential `ok` lines within a 300 s window before external timeout fired; same 65 tests with `--no-isolation` complete much faster. Also: `timeout N m test ... 2>&1 \| tee file` interacts badly with timeout's SIGTERM — m-cli's stdio appears block-buffered when stdout is a pipe. | STDJSONTST and other high-test-count suites tagged with isolation-throughput notes; recommend `--no-isolation` for >50-test suites in module docs. m-cli improvements open: (1) line-buffer the TAP/text formatter (or pass `-u` / call `sys.stdout.flush()`); (2) document the `--no-isolation` perf characteristic in `m test --help`. | open |
| 2026-05-06 | docs | m-stdlib / YottaDB | `.x(SUBS)` syntax — passing a subscripted local by reference | Surfaced re-enabling the deferred STDLOG-JSON-emission tests after the STDASSERT.raises ZGOTO fix. STDJSON's recursive descent (`$$encodeArray` / `$$encodeObject` / `parseObject` / `parseArray`) was passing `node(k)` by reference, which YottaDB rejects at compile time with `%YDB-E-COMMAORRPAREXP, Comma or right parenthesis expected but not found`. Standard ANSI M permits passing only whole locals (`.x`) by reference, not subscripts (`.x(SUBS)`). YottaDB does not extend that. | STDJSON refactored to **merge-then-pass** idiom (`kill tmp  merge tmp=node(k)  do f(.tmp)` for read-only subtree access; `kill tmp  do f(.tmp)  merge node(k)=tmp` for write-back) in commit `c3a0880`. STDLOG-JSON tests followed in `fb48f39`. Idiom documented for future modules; no m-stdlib code remains affected. | **documented 2026-05-07** — not a bug, documented design |
| 2026-05-06 | P2 | vista-meta / YottaDB | shared YDB container leaks `mumps` processes after STDJSONTST runs, blocks subsequent `m test` invocations | Reproduced 2026-05-06 attempting to re-run STDSEEDTST: every `m test tests/STDSEEDTST.m` returned `rc=143/Terminated` from the SSH transport before any TAP output reached stdout, even with explicit 540 s timeouts. SSH multiplex master also hits `Session open refused by peer` after the leak (sshd `MaxSessions` cap). Recovery requires container restart or `mupip rundown -region "*"` from inside the container. **Container restart caveat (2026-05-07)**: `make restart` doesn't always re-wire `/etc/vehu_authorized_keys` → `/home/vehu/.ssh/authorized_keys`, and a non-graceful prior shutdown leaves the DB in `%YDB-E-REQRUNDOWN`. | m-stdlib developers carry a documented recovery runbook (host-side `docker exec` to reinstall authorized_keys; `mupip rundown -region "*"` from inside the container). vista-meta improvements open: detect leaked `mumps` between SSH sessions and either rundown automatically or surface count to m-cli runner; m-cli could kill its multiplex master and reconnect on `Session open refused` rather than burning the timeout. m-test-engine (the lighter Docker substrate) avoids this entirely — preferred default for non-VistA work. | open (in vista-meta); avoided in default m-test-engine path |
| 2026-05-07 | P2 | m-stdlib / vista-meta | STDSEEDTST `tLoadFilerErrorPropagatesEcode` (test #12) timeout when `$ETRAP` fires deep in `walk^STDSEED` while a non-principal SEQ device is current | After a clean container reset, STDSEEDTST still TIMEOUT'd after 13/13 ok lines. Root cause: when `$ETRAP` fires deep in `walk^STDSEED` while a non-principal SEQ device (the OPENed TSV fixture) is current, the trap's ZGOTO unwind hangs because the device-context state isn't released. (Initial diagnosis was off: TAP plan miscount made it look like test #14 was the hanging test.) | `raises^STDASSERT`'s `$ETRAP` string now prepends `use $principal` — restoring the principal device before the unwind lets ZGOTO complete. Verified by re-enabling all 6 STDSEED loadJson tests; STDSEEDTST 35/35 green. Safe for every other `raises^STDASSERT` consumer because XECUTE's caller is always at `$principal` at trap-fire time — `use $principal` is a no-op when the current device already is principal. | **resolved 2026-05-07** |
| 2026-05-07 | P2 | m-cli | `m fmt` mangles `$ZF` into `$zfind` (and `$ZCALL` into `$zcstatusall`) | Surfaced landing H1 STDCRYPTO — first Phase 3 callout. `$ZF("symname",args...)` is YDB's legacy external-call intrinsic. Writing `set rc=$ZF("crypto_sha256",data,.out)` triggers the PostToolUse `m fmt` hook, which silently rewrites the line as `set rc=$zfind("crypto_sha256",data,.out)` — and `$zfind` is a string-search intrinsic with completely unrelated semantics. Same shape on `$ZCALL` (mangles to `$zcstatusall`). Longest-prefix abbreviation-expansion table matches `$ZF` against `$ZFIND`. | STDCRYPTO dispatches every $ZF call through an XECUTE'd command string built at runtime (extra helper layer `dispatch3` / `dispatch4` plus `m-lint: disable-file=M-MOD-036`). Cost: pollutes every Phase 3 callout module's M side. STDCOMPRESS / STDHTTP inherited the same indirection. m-cli fix open: extend the abbreviation-expansion table to include `$ZF` / `$ZCALL` as canonical-already names, OR require the full token to be a known short form. | open |
| 2026-05-07 | P2 | m-cli | `m fmt` mangles `$zgetenv` into `$zgbldiretenv` | Surfaced landing L17 STDOS. Writing `quit $zgetenv(name)` triggers the `m fmt` hook to silently rewrite as `quit $zgbldiretenv(name)`. The mangling shape: `$zget` → `$zgbldir` matches the longest known intrinsic starting with `$zg`; trailing `etenv` kept verbatim, yielding `$zgbldir` + `etenv` = `$zgbldiretenv` (not a valid YDB intrinsic). The routine loads but the call site fails at runtime. | STDOS uses `$ZTRNLNM(name)` — the VAX/VMS-equivalent intrinsic that YDB also supports, functionally equivalent for env-var lookup. fmt leaves `$ztrnlnm` untouched (no longer-prefix intrinsic starts with `$ztr`). All four STDOS labels (`env` / `cwd` / `user` / `hostname`) use it. m-cli fix open: extend the abbreviation table OR require exact-prefix match (option 2 is safer — silent expansion of any prefix-overlapping token is a foot-gun). | open |
| 2026-05-05 | docs | YottaDB / m-stdlib | `tstart`/`trollback` must balance per routine frame (TPQUIT) | **Not a YDB defect — documented design.** YDB enforces that a routine cannot `quit` with an unbalanced `tstart` (raises `%YDB-E-TPQUIT`). Python-style fixture API (`setup() opens; teardown() closes`) is structurally impossible — matching `trollback`/`tcommit` must live in the same routine frame as the `tstart`. Hit writing L8 STDFIX: the orchestration-plan §6.4 sketch described `SETUP^STDFIX(tag)` / `TEARDOWN^STDFIX(tag)` as separate procedures, which compile fine but blow up at runtime. | STDFIX exposes only one-shot wrappers (`with(tag,code)` / `invoke(tag,code)`) that open AND close the scope in the same frame. Runner protocol consumes `with`/`invoke` instead of `setup`/`teardown`. orchestration-plan §6.4 should be updated to match the as-built API. | **documented (design-time)** |
